\chapter{Selecting the Best Sigma Parameter}
\label{ch:sigma_selection}

\section{Overview}

The upsampling factor $\sigma$ is among the most impactful parameters in FINUFFT. It determines the size of the internal FFT grid relative to the output grid, and thus directly controls the memory footprint, cache behaviour, and compute cost of the transform. This chapter investigates how to select $\sigma$ optimally for a given input configuration and accuracy requirement.

\section{The Role of $\sigma$ in FINUFFT}

The upsampling factor $\sigma$ controls the size of the internal uniform FFT grid, which has $\sigma N$ points per dimension relative to the $N$ requested output modes. Increasing $\sigma$ enlarges the FFT grid, raising FFT cost, but simultaneously relaxes the accuracy requirement on the spreading kernel, permitting a smaller kernel half-width $w$. Because $w$ decreases and the grid size increases as $\sigma$ grows, there exists a value of $\sigma$ that minimises total transform cost for a given problem size, dimension, and tolerance.

\subsection{Estimating Spreading Cost}

The spreading step accumulates contributions from each of the $M$ non-uniform source points onto a $w^d$ neighbourhood of grid cells. The minimum kernel half-width $w$ required to achieve tolerance $\varepsilon$ at upsampling $\sigma$ in $d$ dimensions is
\begin{equation}
    w = \left\lceil \frac{\log\!\left(\dfrac{0.18 \cdot 1.4^{d-1}}{\varepsilon}\right)}{\pi \sqrt{1 - 1/\sigma}} + 1 \right\rceil.
    \label{eq:kernel_width}
\end{equation}
The theoretical spreading cost is therefore proportional to $M w^d$. In practice the inner loop is vectorised, so the per-point work scales as $\lceil w / W_{\mathrm{simd}} \rceil \cdot w^{d-1}$, where $W_{\mathrm{simd}}$ is the SIMD vector width in grid points.

\subsection{Estimating FFT Cost}

The FFT operates on the upsampled grid of total size $\tilde{N} = \mathrm{next235}(\sigma N)^d$, where $\mathrm{next235}(x)$ denotes the smallest integer $\geq x$ whose only prime factors are $2$, $3$, and $5$. Rounding up to such a highly composite number ensures the FFT can be decomposed into small radix-2, -3, and -5 butterflies, which most FFT libraries handle efficiently. The grid contributes $\mathcal{O}(\tilde{N} \log \tilde{N})$ operations, though the precise constant depends on the chosen FFT backend and cache effects, so this estimate is best treated as an asymptotic guide.

\section{Optimal $\sigma$ as a Function of Input}

The total transform cost is the sum of both contributions,
\begin{equation}
    T(\sigma) \;\approx\; M\, w(\sigma,\varepsilon)^d \;+\; \gamma\, \tilde{N}(\sigma)\log \tilde{N}(\sigma),
    \label{eq:cost_model}
\end{equation}
where $\gamma = C_{\mathrm{FFT}} / C_{\mathrm{spread}}$ is a single hardware-dependent constant expressing the relative cost of one FFT butterfly versus one spreading operation. Since only the ratio matters for finding the optimal $\sigma$, the overall scale cancels.

\subsection{Fitting the Balancing Constant $\gamma$}

The balancing constant $\gamma$ is the only free parameter in the cost model~\eqref{eq:cost_model}; it encodes the relative throughput of the FFT and spreading kernels on a given machine. Because this ratio is hardware-dependent and not analytically available, $\gamma$ must be calibrated empirically.

Seven parameter combinations were selected for the calibration benchmarks, covering a representative cross-section of real-world workloads. The combinations were drawn from FINUFFT's GitHub issue tracker and discussion threads, where users report the tolerance, dimension, and floating-point precision that characterise their applications. The chosen combinations are listed in Table~\ref{tab:gamma_configs}.

For each combination, the transform was benchmarked at four non-uniform point densities $\rho \in \{0.5, 1, 10, 100\}$, where $\rho = M / N$ is the ratio of non-uniform source points to output modes. This range spans regimes from sparse inputs, where spreading dominates, to dense inputs, where the FFT cost becomes comparable. The transform execution time was measured with Google Benchmark for upsampling factors $\sigma \in [1.25, 2.00]$ in steps of $0.05$, yielding a timing curve per density per configuration. A representative selection of these curves is shown in Figure~\ref{fig:all_results}.
                            


Due to limited computational resources, each $(\sigma, \rho)$ combination was measured only a small number of times. This made the raw timing curves susceptible to transient outliers caused by OS scheduling, cache interference, and other system noise. To reduce this effect, a 3-point median filter was applied to each timing curve along the $\sigma$ axis before any further analysis. Median smoothing was preferred over averaging because it suppresses isolated spikes without distorting the location of the minimum, which is the quantity of interest. Figure~\ref{fig:smoothing} illustrates the effect: the raw curve exhibits several sharp spikes, while the smoothed curve preserves the overall shape and the position for the optimal $\sigma$ fitting.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.7\textwidth]{figures/smoothing.pdf}
    \caption{Effect of 3-point median smoothing on a raw timing curve.}
    \label{fig:smoothing}
\end{figure}

Each benchmark run recorded three quantities as a function of $\sigma$: the spreading complexity $S(\sigma) = M w(\sigma, \varepsilon)^d$, the FFT complexity $F(\sigma) = \tilde{N}(\sigma) \log \tilde{N}(\sigma)$, and the smoothed measured time $\hat{T}(\sigma)$. All three are overlaid in Figure~\ref{fig:all_results}. The goal of the fitting procedure is to find the value of $\gamma$ for which the model sum $S(\sigma) + \gamma F(\sigma)$ attains its minimum at the same $\sigma$ as the real measurement.

The loss for a single $(\text{config},\rho)$ pair is defined as follows. Let $\sigma^*_{\text{model}} = \arg\min_\sigma \bigl[S(\sigma) + \gamma F(\sigma)\bigr]$ be the $\sigma$ selected by the model, and let $\sigma^*_{\text{meas}} = \arg\min_\sigma \hat{T}(\sigma)$ be the empirically optimal $\sigma$. The error is the difference in measured time between these two points,
\begin{equation}
    \ell(\gamma) = \hat{T}(\sigma^*_{\text{model}}) - \hat{T}(\sigma^*_{\text{meas}}),
    \label{eq:gamma_loss}
\end{equation}
which is zero when the model chooses the true optimum and positive when it selects a suboptimal $\sigma$. Figure~\ref{fig:gamma_error} illustrates this loss for a single density before any tuning, i.e.\ with $\gamma = 1$, showing the duration between the model-predicted and empirically optimal $\sigma$.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.7\textwidth]{figures/error.pdf}
    \caption{Error between the real time measurement and the complexity prediction as a function of the upsampling factor $\sigma$.}
    \label{fig:gamma_error}
\end{figure}

To find the optimal $\gamma$, the total loss (summed in squares over all configurations and densities) is minimised by a two-stage sweep: a coarse logarithmic scan over $[10^{-3}, 10^{3}]$ to locate the approximate minimum, followed by a fine linear scan in a neighbourhood of that point.
The fitting procedure yields $\gamma = 16.075$. Figure~\ref{fig:all_results} shows the resulting predicted complexity $S(\sigma) + \gamma F(\sigma)$ overlaid on the timing curves for some configurations and density, confirming that the model minimum is close to the measured minimum across the benchmark set.
